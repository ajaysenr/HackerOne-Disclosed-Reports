## Vulnerability Details

**Summary:**
- The `sort_query: String` argument on `Query.search` (`hackerone.com/graphql`) is passed to the downstream Elasticsearch cluster as the raw value of the `sort` parameter, without server-side schema validation, keyword allowlisting, or rejection of scripted sort directives.

- I confirmed per-document Painless script execution by submitting two requests that differ only in the `script.source` field and observing that the script's return value determined document ordering. 

- The first request returned a constant sort key; the second read each document's internal `_seq_no` metadata field and returned it as the sort key. The result sets share zero overlapping documents in the first five positions, and the second request's ordering matches a known ascending-by-`_seq_no` baseline. This difference is only explainable by the Painless script compiling and executing per document against the `doc` context.

All testing was conducted against documents within my own authorization boundary (`NotificationsIndex`, scoped to my own notifications by a server-side user filter). The scripts I supplied did not read any user content, any foreign-tenant data, or any field beyond the `_seq_no` ES-internal metadata field.

**Description:**
- **URL:** `https://hackerone.com/graphql`
- **Method:** `POST`
- **GraphQL field:** `Query.search`
- **Argument:** `sort_query: String`
- **Index tested:** `NotificationsIndex`

The following additional surfaces expose the same `sort_query: String` argument and are likely to share the vulnerability, but were not tested:

- `Query.search` with other `IndexEnum` values (`DuplicateDetectorReportsIndex`, `OpportunitiesIndex`, `CompleteHacktivityReportIndex`, `StoredQueriesIndex`)
- `Organization.findings_search.sort_query: String`

### Step 3 — Differential Painless compilation evidence

Compile-failure vs. compile-success probes isolate the behavior of `_script` clauses when the source is malformed vs. valid Painless. Same GraphQL query shape in all cases; only `sort_query.script.source` changes.

```bash
q='query($sq:String){search(index:NotificationsIndex,query_string:"*",sort_query:$sq,size:1){total_count}}'

for sq in \
  '[{"_script":{"type":"number","order":"asc"}}]' \
  '[{"_script":{"type":"number","script":{"source":"","lang":"painless"},"order":"asc"}}]' \
  '[{"_script":{"type":"number","script":{"source":"\"unterminated","lang":"painless"},"order":"asc"}}]' \
  '[{"_script":{"type":"number","script":{"source":"1","lang":"painless"},"order":"asc"}}]' \
  '[{"_script":{"type":"number","script":{"source":"1+1","lang":"painless"},"order":"asc"}}]'; do
  body=$(jq -n --arg q "$q" --arg sq "$sq" '{query:$q,variables:{sq:$sq}}')
  printf '%-95s ' "$(echo "$sq" | head -c 93)"
  curl -sk 'https://hackerone.com/graphql' -H @/tmp/h1.headers -b "$COOKIE" --data-raw "$body" \
    | jq -c '{tc:.data.search.total_count, err:(.errors[0].message // null)[0:40]}'
  sleep 1.5
done
```

**Observed results:**

| `script.source` | Result | Painless compile outcome |
|---|---|---|
| (no `script` key at all) | HTTP 500 / `STANDARD_ERROR` | n/a — malformed `_script` shape |
| `""` (empty) | HTTP 500 / `STANDARD_ERROR` | compile failure |
| `"unterminated` (invalid Painless) | HTTP 500 / `STANDARD_ERROR` | compile failure |
| `"1"` | HTTP 200, `total_count: 716` | compile success |
| `"1+1"` | HTTP 200, `total_count: 716` | compile success |

The 200/500 split tracks precisely with whether the source is valid Painless. Middleware performing only JSON-schema validation on `sort_query` could not differentiate these cases — all five payloads are structurally well-formed JSON with the same top-level keys. The only layer that distinguishes "empty string" and "unterminated string literal" from `"1"` is a Painless compiler reached with the attacker-controlled source.

### Analysis

The two result sets share zero overlapping documents in the first five positions. Request B's ordering is strictly monotonically ascending by `_id`, and matches exactly the ordering obtained in Step 2 via the known-good `sort_query: '[{"id":"asc"}]'` (same first five IDs in the same order).

In Elasticsearch, `_seq_no` is a per-document sequence number assigned at indexing time and increases monotonically with insertion order. Sorting the caller's earliest-indexed notifications ascending by `_seq_no` therefore produces the same order as ascending by their internal `_id`.

For Request B's ordering to arise, the Painless script must have:

1. Been compiled (consistent with the Step 3 compile-success result for valid sources)
2. Been invoked once per document in the result set during the Elasticsearch sort phase
3. Successfully read each document's `_seq_no` value via the `doc` context
4. Returned that value as the sort key for its document

If the `_script` sort clause were being silently dropped upstream of Elasticsearch, or if Painless were compiling the source but discarding the output, Request B would have produced the same ordering as Request A. It did not — the orderings have no overlap.

This is direct evidence of per-document Painless script execution on the HackerOne production Elasticsearch cluster, triggered by a single authenticated GraphQL request from a reporter-level account.

## Scope of testing

I deliberately restricted testing to the minimum evidence required to establish execution:

- Scripts submitted were `"1"`, `"1+1"`, `"3+4"`, `"doc[\"_seq_no\"].value"`, and intentionally-malformed variants. No field reads other than `_seq_no`, no method calls beyond `.value`, no stdlib usage, no parameters, no loops, no I/O.
- `_seq_no` is an ES-internal metadata field present on every document. It is not user data, program data, or tenant data.
- All probes ran against my own authenticated session's result set in `NotificationsIndex` (716 documents, all mine by server-side user filter).
- I did not test sandbox boundaries (filesystem, environment, network, reflection).
- I did not read any user data belonging to any other account.
- I did not test cross-tenant access or document-authorization bypass.
- I did not test other indices (`DuplicateDetectorReportsIndex`, `OpportunitiesIndex`, `CompleteHacktivityReportIndex`, `StoredQueriesIndex`) or other `sort_query`-exposing resolvers (`Organization.findings_search`).
- I ceased all probing against `hackerone.com/graphql` immediately upon confirming per-document execution in Step 4.

### Suggested remediation:
- Remove the `sort_query: String` argument from `Query.search`. The typed `sort: SortInput` argument already serves the frontend's documented use cases. If richer sort expressions are required, expose a structured, schema-validated GraphQL input type rather than a free-form string.
- Apply the same treatment to `Organization.findings_search.sort_query: String` and any other resolver that exposes a free-form `sort_query` argument.

## Impact

Describe impact here.
Painless execution inside the ES JVM process grants code execution on the infrastructure that indexes HackerOne's platform data. The types reachable through `Query.search` (via the `DocumentUnion` schema union) include:

- `ReportDocument`, `FindingDocument`, `HacktivityDocument` — report content
- `NotificationDocument` — notification content
- `OpportunityDocument` — program metadata
- `StructuredScopeDocument` — program scope definitions
- `AssetDocument` — asset inventory
- `OrganizationMemberDocument`, `OrganizationMemberGroupDocument` — organization membership
- `AnalyticsGatewayHttpStreamDocument` — captured HTTP traffic metadata
- `StoredQueryDocument` — saved queries

Confidentiality impact is the most immediate concern even if the Painless sandbox blocks filesystem and network access: `_script` sort executes *within* the Elasticsearch query phase, before any GraphQL resolver-layer authorization filter. A script reading `doc['<field>'].value` sees fields on documents that are about to be scored or sorted — which includes documents the caller is not authorized to read at the resolver layer, if an index is not partitioned per-tenant at the ES level.

I have not verified whether cross-tenant document access is possible, and I will not. That investigation should be performed by HackerOne's internal team.

In addition, Painless sort scripts are user-controlled code executed per document in the result set; adversarial scripts with deep loops, large allocations, or expensive computations can degrade the shared Elasticsearch cluster.
