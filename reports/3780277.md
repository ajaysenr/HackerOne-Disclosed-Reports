## Vulnerability Details

**Summary:** Typically, when an adversary gains access to stolen AWS IAM credentials they will [frequently](https://sysdig.com/blog/scarleteel-2-0/) test those credentials to see what access they have. They do this by performing API calls and seeing which succeed and which fail. There are even automated [tools](https://github.com/andresriancho/enumerate-iam) to make this process easier. For defenders and security professionals, this behavior serves as a golden opportunity for detection as it likely involves generating a large number of failed API call attempts. If an adversary could enumerate permissions without logging to CloudTrail, they could perform this activity invisibly.

There are many categories of CloudTrail bypass. The specific variant we will be focused on in this report has been referred to as "non-production endpoint permission enumeration CloudTrail bypass". If you would like to learn more about it, you can find more details [here](https://securitylabs.datadoghq.com/articles/non-production-endpoints-as-an-attack-surface-in-aws/#silent-permission-enumeration).

**We have found 23 non-production endpoints for the S3 Tables service which could be used with standard IAM credentials, and do not log the tested s3tables operation to CloudTrail.** While it is good that these results do not prove access to customer partition data, the endpoints can still be used for permission enumeration without logging to CloudTrail.

AWS has previously [stated](https://securitylabs.datadoghq.com/articles/non-production-endpoints-as-an-attack-surface-in-aws/#the-response-from-aws) that this type of vulnerability should be reported. Specifically, "For isolated non-production endpoints that do not log to CloudTrail but are otherwise callable with normal credentials and exhibit normal IAM permission behavior, AWS considers the CloudTrail logging bypass of such endpoints also to be a security issue. If you find an API or APIs on an endpoint with these characteristics, please contact the AWS Security Team at ████████".

**Description:**

Automated validation tested the candidate endpoint with two principals:

- `admin`: privileged role used to observe authorized candidate endpoint behavior.
- `noperm`: no-permission role used to confirm the candidate endpoint exhibits useful IAM authorization behavior.

For each endpoint in this report, the candidate endpoint call did not generate a matching CloudTrail event for either principal, and the candidate endpoint responses differed between the privileged and no-permission principals.

## Steps To Reproduce:

Using a sufficiently privileged IAM user or role, perform the following AWS CLI operation against one of the affected candidate endpoints:

```bash
aws s3tables list-table-buckets --region us-west-2 --cli-connect-timeout 10 --cli-read-timeout 10 --endpoint-url https://s3tables.us-west-2.████████.people.aws.dev --no-verify-ssl
```

Wait 10-15 minutes and confirm that the request does not generate a matching CloudTrail event. Next, perform the same candidate endpoint call with a principal that does not have permission for the target action:

```bash
aws s3tables list-table-buckets --region us-west-2 --cli-connect-timeout 10 --cli-read-timeout 10 --endpoint-url https://s3tables.us-west-2.████████.people.aws.dev --no-verify-ssl
```

The second request also does not generate a matching CloudTrail event. An adversary can perform these calls and, depending on the response of the API, determine if an identity they have compromised does or does not have permission to perform the operation.

Additionally, depending on if a principal does or does not have permission to perform the action, the response from the affected endpoint will be different. Example evidence from the validation run is below.

### Example: `list-table-buckets` against `s3tables.us-west-2.████████.people.aws.dev`

CloudTrail lookup for the `admin` candidate endpoint call:

- region `us-east-1`: matched_any=`False`, matched_marker=`False`, events=`0`, sources=`none`
- region `us-west-2`: matched_any=`False`, matched_marker=`False`, events=`0`, sources=`none`

CloudTrail lookup for the `noperm` candidate endpoint call:

- region `us-east-1`: matched_any=`False`, matched_marker=`False`, events=`0`, sources=`none`
- region `us-west-2`: matched_any=`False`, matched_marker=`False`, events=`0`, sources=`none`

`admin` candidate response:

```text
{
    "tableBuckets": []
}
```

`noperm` candidate response:

```text
An error occurred (AccessDeniedException) when calling the ListTableBuckets operation: User: arn:aws:sts::████████:assumed-role/noperm/vm-noperm-20260601T181312Z is not authorized to perform: s3tables:ListTableBuckets on resource: arn:aws:s3tables:us-west-2:████████:bucket/* because no identity-based policy allows the s3tables:ListTableBuckets action
```
