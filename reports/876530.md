## Vulnerability Details

## Summary:

In monerod, there is an outgoing send queue for each p2p connection to other peers. When this send queue is full, the thread sleeps for approximately 6 seconds and then terminates the connection.

It is possible for an attacker to request enough objects to fill this queue repeatedly, causing all (default = 10) of the p2p threads to sleep.

Additionally, a peer can request objects before performing a handshake, allowing the attacker to bypass any limit set on the number of incoming connections.

This also uses up a lot of memory, causing further issues on memory-constrained systems.

This is related to report #506595


## Releases Affected:

 All monero versions, including v0.15.0.5

## Steps To Reproduce:

- Attacker opens many connections and sends simultaneously sends many NOTIFY_REQUEST_GET_OBJECTS (500 blocks) to <target node>, with specially chosen blocks which can fill the queue
For each request:
    - <target node> tries to respond, and fills m_send_que in abstract_tcp_server2.inl
    - The P2P thread in <target node> then sleeps for approximately 6 seconds
With enough well-timed requests, this will cause all the P2P threads in <target node> to sleep.


This can be trivially triggered for a single P2P thread by syncing a monerod instance (v0.15.0.5) with the arguments --block-sync-size=500 --limit-rate-down=500 --add-exclusive-node <target node>. Once it reaches a height at which the size of the next 500 blocks is greater than the m_send_que in <target node>, the P2P thread in the target node will sleep for 6 seconds and then drop the connection. This has been tested with a blockchain synced up to height 1675605 (other heights will work).



## Quick fix

- Check that a handshake has completed before sending blocks
- Further limit the maximum number of NOTIFY_REQUEST_GET_OBJECTS
- Remove the sleep from the P2P thread when the send queue is full
