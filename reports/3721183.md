## Vulnerability Details

## Summary:
I found an incomplete-fix variant of CVE-2026-5545 in curl/libcurl 8.20.0.

libcurl 8.20.0 still wrongly reuses an HTTP Negotiate-authenticated connection for a later request to the same host even when the requested service principal changes via `CURLOPT_SERVICE_NAME` / `--service-name`.

In my reproduction, request 1 authenticates a persistent HTTP connection for service principal `svcA`. Request 2 then targets `svcB` on the same host, but libcurl reuses the already authenticated connection, sends no new `Authorization` header for `svcB`, and performs no new Negotiate handshake for `svcB`. The server still serves the `svcB` request under the auth context previously established for `svcA`.

This is wrong authenticated connection reuse across changed service principals.

This appears to be an incomplete-fix variant of CVE-2026-5545, which addressed wrong reuse of HTTP Negotiate-authenticated connections but did not cover service-principal changes in the reproduced path.

## Root cause notes
The requested service name participates in Negotiate auth context creation, but the HTTP Negotiate connection reuse checks in the reproduced path do not appear to treat the service name as part of the authenticated connection identity.

Relevant code paths in current source include:

* `lib/http_negotiate.c` using `STRING_SERVICE_NAME` / `STRING_PROXY_SERVICE_NAME`
* `lib/http_negotiate.c` passing the selected service into the SPNEGO auth path
* `lib/url.c` reuse matching for authenticated connections not appearing to account for changed service principals in the reproduced case

This allows a connection authenticated for service principal A to be reused for service principal B on the same host.

## Impact

## Summary:
This is an authentication-target confusion issue caused by wrong authenticated connection reuse.

A request intended for service principal B can be sent over a connection already authenticated for service principal A on the same host, with no new Negotiate handshake and no new `Authorization` header for B.

In the attached reproduction, the `/svcB` request succeeds under the previously established `svcA` authenticated connection state, while a fresh request to `/svcB` requires its own Negotiate exchange and yields different behavior.

This breaks the auth-target boundary enforced by service principals and can cause requests for one target service to execute under another service principal's authenticated context.

This is materially similar to the already accepted class of curl HTTP Negotiate connection reuse vulnerabilities and is best understood as an incomplete-fix / variant of CVE-2026-5545.
