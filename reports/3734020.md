## Vulnerability Details

A malicious HTTPS-on-HTTP/2 proxy can grow a libcurl client's resident set without bound during the CONNECT phase by streaming 1xx informational responses. The CVE-2023-38039 cap (`MAX_HTTP_RESP_HEADER_SIZE`, 300 KiB, enforced through `Curl_bump_headersize`) is not applied on the HTTP/2 proxy path. The HTTP/2 proxy filter was introduced in commit `4ae2d9f24d` (2023-04-06, "proxy: http2 proxy tunnel implementation") and the `:status` тЖТ new-`http_resp` тЖТ `prev`-chain pattern with no depth cap has been there since.

The short version: every `:status` pseudo-header on the proxy CONNECT stream goes through `proxy_h2_on_header` (`lib/cf-h2-proxy.c:558-577`), which allocates a fresh `struct http_resp` per status and chains it via `resp->prev`. The tunnel state machine only considers the CONNECT response "final" when it sees a non-1xx status (`lib/cf-h2-proxy.c:506-509`), so a peer that streams 1xx forever keeps the accumulator alive forever. There is no chain-depth cap, and `Curl_bump_headersize` is never called from `cf-h2-proxy.c`.

`Curl_http_resp_make` (`lib/http.c:4973-4998`) init's both `headers` and `trailers` `dynhds` with `DYN_HTTP_REQUEST` = 1 MiB. That's a per-entry ceiling тАФ it does not chain. An attacker that fills each entry just under the per-`dynhds` byte cap and then sends a fresh `:status` to start a new `http_resp` gets ~1 MiB per chain link, no upper bound. Even the bare-`:status` spammer mode allocates ~1.4 KiB per frame against ~3-5 bytes on the wire (compressed HPACK), so roughly 1000├Ч amplification.
