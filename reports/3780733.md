## Vulnerability Details

## Summary:
Curl calls getaddrinfo to resolve a domain's addresses, however glibc will continue though the domain search list to find data even if it gets a NODATA response. When using AF_UNSPEC in the ai_hints, this search will stop at the first domain with either an A or AAAA record, however when using AF_INET or AF_INET6, the search will continue until an address of the specified type is found. This was encountered by someone who had a wildcard subdomain for one of the domains in their search list (https://discourse.nixos.org/t/0a000410-ssl-routines-ssl-error-only-in-nix-sandbox/78015/3).

I apologize if glibc is a better location for this report considering their handling of NODATA (not NXDOMAIN) responses (https://sourceware.org/git/?p=glibc.git;a=blob;f=resolv/res_query.c;h=30ace9d06dcd7fe938df29a654643b656725de9f;hb=HEAD#l481).

## Steps To Reproduce:
Disable nscd and try to connect to an IPv4 only host (e.g. github.com) where an AAAA record exists for a domain in the search list (e.g. github.com.home.arpa). If curl can connect to the IPv6 address for github.com.home.arpa, it will prefer it over the IPv4 address for github.com.

## Impact

## Summary:
An attacker in control of part or all of a domain on a computer's domain search list can intercept and block traffic to IPv4 only hosts. This does not affect certificate validation so TLS traffic can't be decrypted.
