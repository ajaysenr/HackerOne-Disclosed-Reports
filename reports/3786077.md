## Vulnerability Details

## Summary:

curl/libcurl appears to allow unauthenticated SOCKS5 negotiation even when the caller explicitly configures username/password-only SOCKS5 authentication. With `--socks5-basic` and SOCKS5 credentials set, curl still advertises both SOCKS5 method `0x00` (no authentication) and `0x02` (username/password). If the proxy selects `0x00`, curl proceeds with the transfer without performing username/password authentication. This appears to contradict the documented `CURLOPT_SOCKS5_AUTH` behaviour, where `CURLAUTH_BASIC` allows username/password authentication and `CURLAUTH_NONE` is the flag that allows no authentication. The impact is a caller-policy bypass: an application or operator may believe SOCKS5 username/password authentication is required, while the transfer can still complete through unauthenticated SOCKS5.

## Steps To Reproduce:

I used a minimal local SOCKS5 proxy fixture that records curl’s initial SOCKS5 greeting and then chooses which authentication method to select.

The curl invocation used SOCKS5 username/password credentials and requested username/password SOCKS5 authentication with:

```text
--socks5-basic
```

The relevant expectation is:

```text
CURLOPT_SOCKS5_AUTH tells libcurl which SOCKS5 authentication methods are allowed.
CURLAUTH_BASIC allows username/password authentication.
CURLAUTH_NONE allows no authentication.
--socks5-basic maps to CURLOPT_SOCKS5_AUTH=CURLAUTH_BASIC.
```

Observed SOCKS5 greeting:

```text
05 02 00 02
```

Interpreted as:

```text
05       SOCKS version 5
02       two methods offered
00       no authentication
02       username/password
```

So curl advertises both no-auth and username/password even though username/password-only SOCKS5 auth was configured.

Case 1: proxy selects no-auth:

```text
Proxy selected: 0x00
Result: curl proceeded to the HTTP request.
Username/password exchange: none.
Transfer: successful.
```

Runtime transcript:

```text
CASE choice=0x00 exit=0 stdout=b'OK' stderr=b'' methods=['0x00', '0x02'] greeting=05020002 auth=None http_seen=True
```

Case 2: proxy selects username/password:

```text
Proxy selected: 0x02
Result: curl performed the expected username/password authentication exchange.
Transfer: successful.
```

Runtime transcript:

```text
CASE choice=0x02 exit=0 stdout=b'OK' stderr=b'' methods=['0x00', '0x02'] greeting=05020002 auth=('user', 'pass') http_seen=True
```

Expected behaviour:

```text
When configured with username/password-only SOCKS5 authentication, curl should not advertise method 0x00 no-auth.
If a proxy selects a method that was not allowed by the caller's SOCKS5 auth configuration, curl should reject the negotiation.
```

Observed behaviour:

```text
curl advertises no-auth and accepts no-auth when selected by the proxy.
```

I can provide the minimal local proxy fixture and full runtime evidence bundle as a seperate attachment if useful.

## Impact

The issue is not that a proxy can choose no-auth. The issue is that curl offers no-auth even after the caller has explicitly asked for username/password SOCKS5 authentication. (!)

As I understand the API, `CURLOPT_SOCKS5_AUTH` is not just a preference hint. It defines which SOCKS5 authentication methods libcurl is allowed to use. `CURLAUTH_BASIC` means username/password. `CURLAUTH_NONE` is the separate option for unauthenticated SOCKS5.

So when a caller uses `--socks5-basic` or `CURLOPT_SOCKS5_AUTH=CURLAUTH_BASIC`, they are making a security decision: do not proceed through unauthenticated SOCKS5.

The observed behaviour breaks that expectation. curl still offers `0x00` no-auth on the wire, and if the proxy selects it, curl continues without the username/password exchange.

This would be a real problem on for systems (like ours) where authenticated proxy use is part of the security model, for example auditability, routing policy, tenant separation, delegation or proxy-side authorisation. In those cases, curl can silently take a weaker path than the caller configured.

Also if this is part of a security policy anywhere it's not doing the secure thing intended.
