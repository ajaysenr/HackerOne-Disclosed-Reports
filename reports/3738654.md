## Description:

### Original vulnerability: direct post-activation PCI queue mutation

The original issue behind CVE-2026-5747 was in the virtio PCI common configuration write path. In the vulnerable code, queue configuration registers were written directly into the live `Queue` object without checking whether the device had already been activated.

Vulnerable code in the old tree:

File: `src/vmm/src/devices/virtio/transport/pci/common_config.rs`  
Lines: `202`, `219`, `314-330`

```rust
0x18 => self.with_queue_mut(queues, |q| q.size = value),
...
0x1c => self.with_queue_mut(queues, |q| {
    if value != 0 {
        q.ready = value == 1;
    }
}),
...
0x20 => self.with_queue_mut(locked_device.queues_mut(), |q| {
    lo(&mut q.desc_table_address, value)
}),
0x24 => self.with_queue_mut(locked_device.queues_mut(), |q| {
    hi(&mut q.desc_table_address, value)
}),
0x28 => self.with_queue_mut(locked_device.queues_mut(), |q| {
    lo(&mut q.avail_ring_address, value)
}),
0x2c => self.with_queue_mut(locked_device.queues_mut(), |q| {
    hi(&mut q.avail_ring_address, value)
}),
0x30 => self.with_queue_mut(locked_device.queues_mut(), |q| {
    lo(&mut q.used_ring_address, value)
}),
0x34 => self.with_queue_mut(locked_device.queues_mut(), |q| {
    hi(&mut q.used_ring_address, value)
}),
```

That was dangerous because queue activation validates queue geometry only once and then caches raw host pointers derived from that geometry.

File: `src/vmm/src/devices/virtio/queue.rs`  
Lines: `342-369`

```rust
pub fn initialize<M: GuestMemory>(&mut self, mem: &M) -> Result<(), QueueError> {
    ...
    self.desc_table_ptr =
        self.get_aligned_slice_ptr(mem, self.desc_table_address, self.desc_table_size(), 16)?;
    self.avail_ring_ptr =
        self.get_aligned_slice_ptr(mem, self.avail_ring_address, self.avail_ring_size(), 2)?;
    self.used_ring_ptr =
        self.get_aligned_slice_ptr(mem, self.used_ring_address, self.used_ring_size(), 4)?;
    Ok(())
}
```

After activation, later queue operations trust `self.size` for indexing, but still dereference the cached pointers. If the guest changes queue fields after activation, Firecracker can perform out-of-bounds virtqueue reads and writes through stale pointers.

### How the fix works

The fix in `1.14.4` / `1.15.1` adds a guard that only permits queue configuration writes while the device is in the virtio setup state `ACKNOWLEDGE | DRIVER | FEATURES_OK` (`0xb`), before `DRIVER_OK` and before backend activation is complete.

Fixed code:

File: `src/vmm/src/devices/virtio/transport/pci/common_config.rs`  
Lines: `237-254`, `272-294`, `396-413`

```rust
fn update_queue_field<F: FnOnce(&mut Queue)>(&mut self, queues: &mut [Queue], f: F) {
    let status = self.driver_status;
    if status == (ACKNOWLEDGE | DRIVER | FEATURES_OK) {
        self.with_queue_mut(queues, f);
    } else {
        warn!(
            "pci: queue config write not allowed in device status {:#x}",
            status
        );
    }
}
...
QUEUE_SIZE => self.update_queue_field(queues, |q| q.size = value),
...
QUEUE_ENABLE => self.update_queue_field(queues, |q| {
    if value != 0 {
        q.ready = value == 1;
    }
}),
...
QUEUE_DESC_LO => self.update_queue_field(locked_device.queues_mut(), |q| {
    lo(&mut q.desc_table_address, value)
}),
...
QUEUE_USED_HI => self.update_queue_field(locked_device.queues_mut(), |q| {
    hi(&mut q.used_ring_address, value)
}),
```

So the intended invariant is:

- queue pointers are initialized once using validated geometry
- once the backend is active, PCI queue fields are no longer writable

### Bypass path: hostile snapshot restore recreates the forbidden state

The bypass exists because snapshot restore reconstructs queue activation state and PCI `driver_status` from different serialized fields, but does not validate that they are consistent with each other.

1. Queue restore rebuilds cached raw pointers whenever the saved virtio device state says the queue is already activated.

File: `src/vmm/src/devices/virtio/persist.rs`  
Lines: `88-111`

```rust
fn restore(
    constructor_args: Self::ConstructorArgs,
    state: &Self::State,
) -> Result<Self, Self::Error> {
    let mut queue = Queue {
        max_size: state.max_size,
        size: state.size,
        ready: state.ready,
        desc_table_address: GuestAddress(state.desc_table),
        avail_ring_address: GuestAddress(state.avail_ring),
        used_ring_address: GuestAddress(state.used_ring),
        ...
    };
    if constructor_args.is_activated {  //--[1]
        queue.initialize(&constructor_args.mem)?;
    }
    Ok(queue)
}
```

2. PCI restore independently trusts `state.device_activated` and re-activates the backend.

File: `src/vmm/src/devices/virtio/transport/pci/device.rs`  
Lines: `437-459`

```rust
let mut virtio_pci_device = VirtioPciDevice {
    ...
    common_config: virtio_common_config,
    ...
    device_activated: Arc::new(AtomicBool::new(state.device_activated)), //--[1]
    ...
};

if state.device_activated {
    virtio_pci_device
        .device
        .lock()
        .expect("Poisoned lock")
        .activate(
            virtio_pci_device.memory.clone(),
            virtio_pci_device.virtio_interrupt.as_ref().unwrap().clone(),
        );
}
```

3. The queue-write gate still checks only `common_config.driver_status`.

File: `src/vmm/src/devices/virtio/transport/pci/common_config.rs`  
Lines: `244-247`

```rust
let status = self.driver_status;
if status == (ACKNOWLEDGE | DRIVER | FEATURES_OK) {
    self.with_queue_mut(queues, f);
}
```

This means a hostile snapshot can restore:

- `VirtioDeviceState.activated = true`
- `VirtioPciDeviceState.device_activated = true`
- `VirtioPciCommonConfigState.driver_status = 0xb`

After restore, Firecracker has:

- an already active backend
- queues whose raw pointers were reinitialized from the saved geometry
- PCI queue writes permitted again because `driver_status == 0xb`

That recreates the same stale-pointer condition that the original fix was meant to prevent.

### Confirmed consequence

Using the hostile restore state above, the guest can mutate the live queue again after restore and revive the old stale used-ring path:

- out-of-bounds avail-ring read relative to the originally validated queue
- out-of-bounds descriptor-table read relative to the originally validated queue
- out-of-bounds used-ring write through the stale `used_ring_ptr`

- PCI `driver_status` is rolled back to `0xb`

### Stack Trace

When the reviewer attaches with GDB at the `restore_loaded_paused` stage and then resumes the VM, the expected backtrace shows the crash in the virtio queue used-ring write path. The crash occurs because the used-ring write goes past guest RAM and lands in an unmapped page.

## Impact

## Summary:

Firecracker is an open source virtualization technology purpose-built for secure multi-tenant container and function workloads. A variant of CVE-2026-5747 exists where, under hostile snapshot restore conditions, a guest can bypass the post-activation virtio PCI queue-write fix by restoring an active device with rolled-back PCI driver_status, reviving stale virtqueue pointer use and enabling out-of-bounds virtqueue access, including a host-crashing out-of-bounds used-ring write.
