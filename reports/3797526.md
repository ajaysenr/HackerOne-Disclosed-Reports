## Vulnerability Details

### Summary

When a user invokes curl with a schemeless URL and `--proto-default sftp` (or `scp`), the tool layer guesses the URL is HTTP and skips setting SSH security options (`CURLOPT_SSH_HOST_PUBLIC_KEY_SHA256`, `CURLOPT_SSH_KNOWNHOSTS`). However libcurl's runtime correctly applies `--proto-default` and connects via SFTP/SCP. The result is that `--hostpubsha256`, `--hostpubmd5`, and `--knownhosts` are silently ignored — curl connects to an unverified SSH host and sends authentication credentials.

A network MITM attacker running a rogue SSH server captures the victim's password or triggers key-based auth against an attacker-controlled host, while curl reports no host key error.

### Affected version

Introduced in commit `f97d372703` (2025-05-15, "tool_operate: move config2setopts to separate file"), first released in **curl 8.14.0**. Present through current master (8.21.0-DEV).

### Root cause

In `src/config2setopts.c`, `url_proto_and_rewrite()` uses `CURLU_GUESS_SCHEME` to determine `use_proto`. For a schemeless URL like `host:2222/path`, it guesses HTTP. The SSH options gate on this guess:

```c
// src/config2setopts.c:190
if(use_proto != proto_scp && use_proto != proto_sftp)
  return CURLE_OK;  // skips setting CURLOPT_SSH_HOST_PUBLIC_KEY_SHA256, CURLOPT_SSH_KNOWNHOSTS, etc.
```

But `CURLOPT_DEFAULT_PROTOCOL` is set later (line ~1053), and libcurl's `lib/url.c:1662` prepends `sftp://` at runtime. So the actual transfer uses SFTP with no host key verification configured.

### Steps To Reproduce

The PoC starts a Paramiko SSH server, then runs curl twice: once with explicit `sftp://` (host key check works) and once with `--proto-default sftp` + schemeless URL (host key check bypassed, password captured).

Prerequisites: `pip install paramiko`. Run from the curl source tree root (`~/curl/curl`).

### Expected behavior

`--hostpubsha256` (and `--knownhosts`) must be applied regardless of whether the SFTP scheme comes from the URL directly or from `--proto-default`. The host key mismatch should abort the connection before any authentication attempt.

### Actual behavior

With `--proto-default sftp` and a schemeless URL, the host key pin is silently discarded. curl proceeds to authenticate against the unverified host, sending the password to whatever SSH server answers at that address.

## Impact

## Summary:
A MITM attacker on the network path intercepts the SSH connection and presents their own host key. Because curl skipped the pin check, the victim's password (or other credential) is sent to the attacker's rogue SSH server (requires explicit command-line passing). The user believes they are protected by `--hostpubsha256` but the protection is silently absent.

This is a credential disclosure to a MITM — the user explicitly requested host verification, curl silently failed to enforce it, and the credential crossed the trust boundary to an unverified endpoint.
