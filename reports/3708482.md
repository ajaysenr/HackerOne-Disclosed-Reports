## Vulnerability Details

I found that `wcurl` does not always keep operands after `--` in a pure URL-data context. The documented way to pass curl options through `wcurl` is `--curl-options`, but a value supplied as a URL operand can still reach the final `curl` command as an option, for example `wcurl -- "--url=file:///..."`. A single URL argument containing a tab is also split by `for url in ${URLS}`, allowing one attacker-controlled URL string to create an additional curl transfer. This matters when a script or service safely quotes an untrusted URL as `wcurl -- "$url"` and expects the value to remain only a URL. The attached local-only PoC shows `direct_result=url_option_executed` and `tab_result=second_transfer_option_executed`. I am not claiming command execution, privilege escalation, or Critical severity; the demonstrated issue is an argument-boundary problem between `wcurl` URL operands and curl options.

## Summary

I found that `wcurl` does not always keep operands after `--` in a pure URL-data context. The documented way to pass curl options through `wcurl` is `--curl-options`, but a value supplied as a URL operand can still reach the final `curl` command as an option, for example `wcurl -- "--url=file:///..."`. This matters when a script or service safely quotes an untrusted URL as `wcurl -- "$url"` and expects the value to remain only a URL. I am not claiming command execution, privilege escalation, or Critical severity; the demonstrated issue is an argument-boundary problem between `wcurl` URL operands and curl options.

## How I analyzed it

I started by checking the intended interface from the local `wcurl --help` and `man wcurl` output. Both describe `--curl-options` as the explicit option-forwarding mechanism, while `<URL>` is described as the URL to download. Based on that interface, I expected arguments after `wcurl --` to remain URL operands, not curl options.

I then inspected the local `/usr/bin/wcurl` script and focused on how it stores URL operands. The script appends each URL to a single string variable named `URLS`, encodes only literal spaces with `sed 's/ /%20/g'`, and later re-expands that string with unquoted shell expansion. That combination is important because shell word splitting can still treat tabs and newlines as delimiters, and leading-dash tokens can still be interpreted by the final `curl` process as options.

The relevant local source lines are:

```sh
# line 238
set -- $URLS

# line 255
for url in ${URLS}; do

# line 263
set -- "$@" ${NEXT_PARAMETER} ${PER_URL_PARAMETERS} ${CURL_NO_CLOBBER} --output "${OUTPUT_PATH}" ${CURL_OPTIONS} "${url}"

# lines 326-332
--)
    shift
    for url in "$@"; do
        # Encode whitespace into %20, since wget supports those URLs.
        newurl=$(printf "%s\n" "${url}" | sed 's/ /%20/g')
        URLS="${URLS} ${newurl}"
    done
    break
    ;;
```

The last `"${url}"` is quoted against shell splitting at that point, but it is still passed to curl as an argv element that starts with `--`. curl then treats it as a curl option rather than as URL data.

## Payloads

Payload 1: a URL operand that starts with a curl option name.

```bash
wcurl -- "--url=file:///tmp/wcurl_argument_injection_poc/source.txt"
```

Payload 2: one quoted shell argument containing a tab, causing `wcurl` to create a second transfer.

```bash
tab_payload=$(printf 'file://%s\t--url=file://%s' "$BENIGN" "$SECRET")
wcurl -- "$tab_payload"
```

Payload 3: a minimal dry-run check showing that a post-`--` operand remains a curl option.

```bash
wcurl --dry-run -- --help
```

## What the PoC proves

First, `wcurl -- "--url=file://$SRC"` places the string after `--`, so it is a `wcurl` URL operand rather than a `wcurl` option. The dry-run shows that the final command still contains `--url=file:///tmp/.../source.txt` as a curl option. The actual run then creates `source.txt` in the download directory, confirming that curl processed the operand as its own `--url` option.

Second, the tab payload is passed to `wcurl` as one quoted shell argument. Despite that, `wcurl` splits it into two transfers internally because `URLS` is later expanded as a shell string. The second half of the original URL string becomes `--url=file://.../secret.txt`, and curl executes it as a second transfer.

## Why I do not think this is intended behavior

The intended curl-option forwarding mechanism is documented as `--curl-options`:

```text
<CURL_OPTIONS>: Any option supported by curl can be set here. This is not used by wcurl; it is instead forwarded to the curl invocation.
```

The URL argument is documented separately as URL data:

```text
<URL>: URL to be downloaded. Anything that is not a parameter is considered an URL. Whitespace is percent-encoded and the URL is passed to curl, which then performs the parsing. May be specified more than once.
```

Because of that separation, I expected an operand after `wcurl --` to remain URL data. The current implementation allows that boundary to be bypassed when the URL starts with a dash or contains tab-delimited content.

## Things I am not claiming

I am not claiming remote code execution. I am not claiming privilege escalation. I am not claiming authentication bypass. I am not claiming arbitrary file overwrite. I am also not claiming a Critical severity issue. The confirmed behavior is that URL data can become curl options or additional curl transfers under an untrusted URL automation scenario.

## Suggested fix direction

The safest direction would be to avoid storing URL operands in a single shell-expanded string. URL operands should be preserved as distinct values, or rejected/encoded if they contain shell IFS/control delimiters such as tab, CR, or LF. The final curl invocation should also place URL data behind an explicit URL boundary such as `--url "$url"`, so a leading-dash URL operand cannot be parsed as a curl option.

## Detailed analysis process

I first treated this as a URL/option-boundary review rather than as a memory-corruption issue. `wcurl` is a shell wrapper around curl, so the important security question was whether data that the caller intends to pass as a URL can cross into an option context before the final `exec curl ...` call.

### 1. Interface expectation

I started from the documented interface:

```text
--curl-options <CURL_OPTIONS>: Specify extra options to be passed when invoking curl.
<URL>: URL to be downloaded. Anything that is not a parameter is considered an URL.
```

That creates a clear boundary: curl options should be passed through `--curl-options`, while URL operands should remain URL data. In particular, a caller using `wcurl -- "$url"` would normally expect `"$url"` to be treated as a URL operand even if the URL string comes from an untrusted source.

### 2. Source review

I then reviewed how `/usr/bin/wcurl` handles URL operands. The relevant parsing block for `--` appends every remaining argument into a single shell string:

```sh
--)
    shift
    for url in "$@"; do
        newurl=$(printf "%s\n" "${url}" | sed 's/ /%20/g')
        URLS="${URLS} ${newurl}"
    done
    break
    ;;
```

Two details stood out here:

1. URL operands are not preserved as separate argv values; they are joined into one string variable.
2. Only literal ASCII space is percent-encoded. Other shell word-splitting delimiters, especially tab and newline, are not encoded.

Later, `wcurl` re-expands that string:

```sh
set -- $URLS
...
for url in ${URLS}; do
    set -- "$@" ${NEXT_PARAMETER} ${PER_URL_PARAMETERS} ${CURL_NO_CLOBBER} --output "${OUTPUT_PATH}" ${CURL_OPTIONS} "${url}"
    NEXT_PARAMETER="--next"
done
```

This means that tab-delimited content inside a single original URL argument can become multiple `url` loop iterations. The final `"${url}"` is quoted at the shell level, which prevents another round of shell splitting there, but it does not prevent curl from interpreting a leading-dash argv element as a curl option.

### 3. Initial hypothesis

From the code review, I formed two testable hypotheses:

1. A URL operand after `--` that starts with a curl option name, such as `--url=file:///...`, may be interpreted by curl as an option.
2. A single quoted URL string containing a tab may be split by `wcurl` into two transfers, and the second part may also be interpreted as a curl option.

I deliberately avoided claiming shell command injection because the final execution path uses `exec "$@"`, not `eval` or `sh -c`. The bug class I tested was argument injection / option boundary confusion, not OS command injection.

### 5. Safe PoC design

For the actual PoC, I used only local `file://` URLs. This avoided any dependency on DNS, remote servers, or network timing, and it made the result easy to verify with marker strings.

I also separated the source directory from the download directory. That was important because otherwise the test could confuse the original source file with the file created by the curl transfer. The PoC writes marker content into source files, runs `wcurl` from a separate download directory, and then verifies whether the expected marker appears in a newly created file under that download directory.

The two marker strings were:

```text
DIRECT_URL_OPTION_EXECUTED
TAB_INJECTED_TRANSFER
```

### 6. Direct option-operand test

The direct test was:

```bash
wcurl -- "--url=file://$SRC"
```

The dry-run showed:

```text

```

The actual run created `source.txt` in the download directory with the marker string:

```text

```

That result shows that the value supplied as a `wcurl` URL operand was processed by curl as its own `--url` option.

### 7. Tab-splitting test

The tab test was:

```bash
tab_payload=$(printf 'file://%s\t--url=file://%s' "$BENIGN" "$SECRET")
wcurl -- "$tab_payload"
```

The important point is that `"$tab_payload"` is one shell argument at the call site. The split happens inside `wcurl`, not in the caller's shell.

The dry-run showed two transfer sections:

```text
file:///tmp/wcurl_argument_injection_poc/benign.txt
--next
...
--url=file:///tmp/wcurl_argument_injection_poc/secret.txt
```

The actual run created the injected second-transfer output:

```text

```

That result confirms that one URL operand can be internally split into an additional curl transfer.

### 8. Negative result I am not using as an impact claim

I also tested a `--config=...` style payload because it is a natural curl option-injection target. In this environment, the dry-run showed that `--config=...` reached curl as an option, but the full marker-file effect did not reproduce reliably because the surrounding `--output` handling affected the transfer/config behavior.

For that reason, I am not using `--config=...` as an impact claim in this report. The confirmed claims are limited to `--url=file://...` option execution and tab-induced extra transfer creation.

### 9. Fix direction

The main fix is to stop storing URL operands as a single shell-expanded string. URL operands should remain distinct values from parsing to final curl invocation. If preserving distinct values is not practical in POSIX shell, `wcurl` should reject or encode all shell IFS/control delimiters, not only literal spaces.

The final curl invocation should also force URL data into URL context. A pattern like this is safer:

```sh
curl ... --output "$OUTPUT_PATH" ... --url "$url"
```

That way, even if `$url` begins with `--`, curl receives it as the argument to `--url` rather than as a new option.

## Impact

An attacker who controls a URL string consumed by automation using `wcurl -- "$url"` can change the meaning of the final curl invocation. The demonstrated impact is that URL data can be interpreted as curl options, and a single quoted URL string containing a tab can create an additional curl transfer. In the local PoC, this allows `file://` input to be read and written into the current download directory as an unintended transfer. If a wrapper service returns downloaded files to the requester, stores them in a trusted location, or processes them as trusted input, this can become a file disclosure or workflow integrity issue in that wrapper. This report does not demonstrate arbitrary code execution, privilege escalation, authentication bypass, or Critical impact by itself.
